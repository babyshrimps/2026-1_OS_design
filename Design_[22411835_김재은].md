## 1. Introduction
 인터넷 쇼핑이 발달함에 따라 다양한 쇼핑몰이 생겨나고 있다. 이에 따라 집에서도 비교적 쉽게, 실시간으로 물품들의 정보를 비교할 수 있게 되었다. 또한 쇼핑 사이트에서 어떤 행사를 하느냐에 따라 할인 비율이 천차만별이기에 사이트에 자주 들어가 현재가를 확인하게 되는데 이 과정에서 사용자는 같은 상품 페이지를 반복해 확인해야 한다.<br>
 Price Trcking Secretary는 이러한 불편함을 줄이기 위해 상품 링크와 목표 가격을 등록하면 시스템이 링크에서 가격을 확인해 목표가격 이하가 됐을 시, 알림을 제공하는 서비스이다.

## 3. Sequence diagram

### 1) Login
<img width="1069" height="863" alt="image" src="https://github.com/user-attachments/assets/5adfa799-3ede-4ee9-b91b-51d2991a9651" />

<br>

시스템 사용자가 서비스에 로그인 하는 과정이다. 사용자가 아이디와 비밀번호를 입력하면 Login 클래스에서 checkUser(inputid, inputPd)를 통해 입력된 정보가 시스템에 저장된 정보와 동일한지 확인한다.
<br>
1. 아이디나 비밀번호가 입력되지 않은 경우, returnLoginResult("입력값 누락") 을 실행해 입력값이 누락되었다는 메세지를 출력한다.
2. 입력 정보가 일반 사용자의 정보와 일치하는 경우, User 클래스에서 사용자 정보를 확인하고 login을 실행해 메인 화면으로 이동한다.
3. 입력 정보가 관리자와 일치하는 경우, Admin 클래스에서 관리자 정보를 확인하고 login을 실행해 관리자 로그 화면으로 이동한다.
4. 입력 아이디, 비밀번호가 저장되어 있는 정보와 일치하지 않는 경우, returnLoginResult("로그인 실패")를 실행하고 로그인 실패 메세지를 출력한다. 
<br>
<br>

### 2) 상품 등록
<img width="758" height="919" alt="image" src="https://github.com/user-attachments/assets/e13af17c-856a-47d7-b3d0-2dfc2c34b340" />
<br>

사용자가 상품을 등록하는 과정이다. <br>
사용자가 상품명, 링크, 목표 가격을 입력하면 productRegister 클래스는 checkInput(productName, productLink, targetPrice)를 통해 입력값이 제대로 되어 있는지 확인한다.
<br>
1. 입력값이 올바른 경우, ProductRegister는 Product 클래스에 addProduct(productName, productLink, targetPrice)를 요청해 새로운 상품 정보를 저장한다. 정보를 저장하면 ProductRegister는 showRegister("상품 등록 완료")를 실행하고 사용자에게 상품 등록 완료 메세지를 출력한다.
2. 상품명, 상품 링크, 목표 가격 중 하나라도 입력되지 않은 경우, ProductRegister가 showRegister("입력값 누락")을 실행 후 입력값 누락 메세지를 출력한다.
3. 상품 링크 형식이 올바르지 않은 경우, showRegister("링크 형식 오류")를 실행하고 링크 형식 오류 메시지를 출력한다.
4. 목표 가격 형식이 올바르지 않은 경우에는 showRegister("가격 입력 오류")를 실행하고 가격 입력 오류 메시지를 출력한다.
5. 입력을 제대로 진행하였으나 정보를 저장하는데 실패한 경우, ProductRegister는 ErrorLog 클래스에 savelog("상품 저장 실패")를 요청해 오류 내용을 저장하고 showRegister("상품 등록 실패")를 실행해 사용자에게 상품 등록 실패 메세지를 출력한다.
<br>
<br>

### 3) 관심 상품 목록 조회
<img width="1017" height="692" alt="image" src="https://github.com/user-attachments/assets/8cdbbce8-2d89-4054-b85a-a4dab89c48ca" />

<br>

사용자가 관심 상품 목록을 조회하는 과정이다. <br>
사용자가 관심 상품 목록 조회를 요청하면 ProductManager 클래스는 showProductList(uwerId)를 실행하여 현재 로그인한 사용자의 상품 목록을 불러온다. 이후 Product 클래스에 사용자 id와 연결된 상품 목록을 요청한다.
<br>
1. 등록된 상품이 있는 경우, Product는 상품 목록을 ProductManager에 반환하고 ProductManager는 사용자에게 관신 상품 목록을 출력한다.
2. 등록된 상품이 없는 경우, 빈 상품 목록을 반환하고 사용자에게 등록된 상품이 없다는 메세지를 출력한다.
3. 상품 목록을 불러오는 과정에서 문제가 생기는 경우, ProductManager는 ErrorLog 클래스에 savelog("상품 목록 조회 실패")를 요청해 오류 내용을 저장하고 사용자에게 상품 목록 조회 실패 메세지를 출력한다.

### 4) 가격 추적
<img width="893" height="793" alt="image" src="https://github.com/user-attachments/assets/db49c260-086e-4ece-ade2-9af4d6513509" />

<br>

가격 추적 기능은 사용자가 실행하는 기능이 아닌 Schedule가 정해진 시간마다 자동으로 실행하는 기능이다.<br>
Scheduler는 startScheduler(checkInterval)를 써서 가격 확인 주기를 관리하고, 정해진 시간이 되면 PriceTracker에 runPriceTracking(productList)를 요청한다. 이후 PriceTracker는 checkPrice(productId)를 실행해 등록된 상품의 현재 가격을 확인한다.
<br>

1. 가격 확인에 성공한 경우, PriceTracker은 mockPriceData(productId)를 통해 모의 가격 데이터를 생성하고 Product 클래스에 updatecurPrice(productId, curPrice)를 요청해 상품의 현재 가격을 갱신한 다. 이 다음 PriceTracker savePriceHistory(productPriceId, price, collectedTime)를 실행하고, PriceHistory 클래스에 savePrice(productPriceId, price, collectedTime)를 요청해 가격 이력을 저장한다. 이후 Product 클래스의 updateStatus(productId, status)를 실행해 상품 상태를 갱신한다.
2. 가격 확인에 실패한 경우, PriceTracker는 Product 클래스에 updateStatus(productId, "수집 실패")를 요청해 상품 상태를 수집 실패로 변경 후 Errorlog 클래스에 savelog("가격 확인 실패")를 요청해 오류 내용을 저장한다.
3. 가격 처리가 끝나면 Scheduler는 updateNextChecktime(nextCheckTime)을 실행해 다음 가격 확인 예정 시간을 갱신한다.
<br>
<br>

### 5) 가격 추이 조회
<img width="1140" height="855" alt="image" src="https://github.com/user-attachments/assets/96c2932e-669b-438d-b093-9c56468f2607" />
<br>

가격 추이 조회 기능은 사용자가 등록한 상품의 가격 변동 흐름을 확인하는 과정으로 사용자가 상품 상세 화면에 접속했을 때 ProductManager 클래스가 showProductDetail(productPriceid)를 요청해 해당 상품의 가격 이력을 불러온다. 그리고 Product 클래스에 상품 정보 요청을 보내서 상품명, 현재 가격, 목표 가격 등의 정보를 가져온다.
<br>

1. 가격 이력이 있는 경우, ProductManager는 PriceHistory 클래스에 getPHistory(productPriceId)를 요청해 해당 상품의 가격 이력을 가져온다. 이후 getTHistory(productPriceId, collectedTime)를 통해 시간 기준으로 정리된 가격 이력을 받아오고, 사용자에게 상품 상세 정보와 가격 추이 그래프를 출력한다.
2. 가격 이력이 없는 경우, PriceHistory는 빈 가격 이력을 반환하고 ProductManager는 사용자에게 가격 이력 없음 메시지를 출력한다.
3. 가격 이력 조회에 실패한 경우, ProductManager는 ErrorLog 클래스에 savelog("가격 이력 조회 실패")를 요청해 오류 내용을 저장한다. 이후 사용자에게 가격 이력 조회 실패 메시지를 출력한다. <br>
<br>

### 6) 알림 내역 조회
<img width="619" height="859" alt="image" src="https://github.com/user-attachments/assets/89c5ed04-487c-44b0-8ae6-ab16f4680a12" />
<br>

알림 내역 조회 기능은 사용자가 목표 가격 도달 알림을 확인하는 과정이다. 사용자가 알림 내역 조회를 요청하면 NotificationManager 클래스는 Notification 클래스에 getNoticeList(NuserId)를 요청해 사용자의 알림 목록을 가져온다.<br>
1. 알림 내역이 있는 경우, Notification 클래스는 알림 목록을 NotificationManager에 반환한다. 이후 NotificationManager는 사용자에게 알림을 출력한다.
2. 알림 내역이 없는 경우, Notification 클래스는 비어있는 알림 목록을 반환한 뒤 NotificationManager는 사용자에게 알림 내역 없음 메세지를 출력한다.
3. 알림 내역 조회에 실패한 경우, NotificationManager는 ErrorLog 클래스에 savelog("알림 내역 조회 실패")를 요청해 오류 내용을 저장한다. 그다음 사용자에게 알림 내역 조회 실패 메세지를 출력한다.
4. 사용자가 모두 읽음 처리를 고른 경우, NotificationManager는 markAllAsRead()를 실행하고 Notification 클래스의 markRead(notificationId)를 통해 알림을 읽음 상태로 전환한다. 전환이 완료되면 사용자에게 읽음 처리된 알림 내역을 출력한다.
<br>
<br>

### 7) 상품 삭제
<img width="571" height="917" alt="image" src="https://github.com/user-attachments/assets/e2670893-57df-45ae-a76b-395a1d43d55e" />
<br>

상품 삭제 기능은 사용자가 가격 추적을 원하지 않는 상품을 삭제하는 과정이다. 사용자가 상품 삭제 버튼을 누르면 ProductManager 클래스는 사용자에게 삭제 확인 메세지를 출력한다. <br>

1. 사용자가 삭제를 확인한 경우, ProductManager는 removeProduct(productId)를 실행하고 Product 클래스에 상품 삭제 요청(productId)을 보낸다. Product에서 상품 삭제가 완료되면 ProductManager는 PriceHistory 클래스에 연결된 가격 이력 삭제를 요청한다. 가격 이력 처리까지 완료되면 사용자에게 상품 삭제 완료 메세지를 출력한다\
2. 사용자가 삭제를 취소한 경우, ProductManager는 상품 정보를 그대로 유지하고 사용자를 이전 화면으로 이동시킨다.
3. 상품 삭제에 실패한 경우, ProductManager은 Product 클래스에게 삭제 실패 결과를 받고 ErrorLog 클래스에 savelog("상품 삭제 실패")를 요청해 오류 내용을 저장한 뒤 이후 사용자에게 상품 삭제 실패 메세지를 출력한다.
4. 이미 삭제된 상품의 경우, Product 클래스가 존재하지 않는 상품이라는 결과를 반환하고 ProductManager는 사용자에게 존재하지 않는 상품 메세지를 출력한다.
   
<br>
<br>

### 8) 목표 가격 변경
<img width="937" height="913" alt="image" src="https://github.com/user-attachments/assets/bdd7cc1c-b0ec-456e-b6f1-578d87de75ba" />
<br>

사용자가 이미 등록한 상품의 목표 가격을 수정하는 기능이다.사용자가 목표 가격 변경을 요청하면 ProductManager 클래스는 changeTargetPrice(productId, targetPrice)를 실행해 입력된 목표 가격이 올바른지 확인한다
<br>

1. 목표 가격 입력값이 올바른 경우, ProductManager가 Product 클래스에 changeTargetPrice(productId, targetPrice)를 요청해 해당 상품의 목표 가격을 변경한다. 목표 가격을 변경하면 사용자에게 가격 변경 완료 메세지를 출력한다.
2. 목표 가격이 숫자가 아닌 경우, ProductManager가 사용자에게 올바른 금액을 입력하라는 메세지를 출력한다
3. 목표 가격이 0원 이하인 경우, ProductManager가 사용자에게 0원보다 큰 금액을 입력하라는 메세지를 출력한다.
4. 목표 가격 저장에 실패한 경우, ProductManager는 Product 클래스로부터 목표 가격 저장 실패 결과를 받고 ErrorLog 클래스에 savelog("목표 가격 변경 실패")를 요청해 오류 내용을 저장한 뒤 사용자에게 목표 가격 변경 실패 메세지를 출력한다.

<br>
<br>

### 9) 오류 및 로그 확인
<img width="494" height="815" alt="image" src="https://github.com/user-attachments/assets/a3850bc5-1d22-48e0-b434-f178d943f2bb" />
<br>

관리자가 시스템에서 발생한 오류, 알림 발송 기록, 시스템 실행 로그를 확인하는 기능이다. 관리자가 오류 및 로그 확인 메뉴를 누르면 Admin 클래스는 ErrorLog()를 실행하고 ErrorLog 클래스에 getLot("오류 로그")를 요청해 저장된 로그들을 불러온다.
<br>

1. 오류 로그가 있는 경우, ErrorLog 클래스는 오류 로그를 Admin 클래스에 반환하고 Admin은 관리자에게 오류 로그를 출력한다.
2. 오류 로그가 없는 경우, ErrorLog 클래스는 빈 오류 로그를 반환하고 Admin은 관리자에게 오류 로그 없음 메세지를 출력한다.
3. 알림 발송 기록이 있는 경우, Admin 클래스는 NoticeLog()를 실행하고 Notification 클래스에 getNoticeList()를 요청한다. 알림 발송 기록이 있는 경우 Notification 클래스가 알림 발송 기록을 확인하고 Admin은 관리자에게 알림 발송 기록을 출력한다.
4. 알림 발송 기록이 없는 경우, Notification 클래스는 빈 알림 기록을 반환하고 Admin은 관리자에게 알림 발송 기록 없음 메세지를 출력한다.
5. 시스템 로그 조회를 성공하는 경우, Admin 클래스는 SystemLog()를 실행한 뒤 관리자에게 시스템 실행 로그를 출력한다.
6. 시스템 로그 조회를 실패하는 경우, Admin 클래스가 SystemLog()를 실행한 뒤 관리자에게 로그 조회 실패 메세지를 출력한다.

<br>

### 10) 알림 전송
<img width="628" height="787" alt="image" src="https://github.com/user-attachments/assets/4b7783e7-90d2-4fc2-9480-c0d291b6beac" />
<br>

등록한 상품의 현재 가격이 사용자가 설정한 목표 가격에 도달했을 때 알림을 전송하는 과정이다. Scheduler는 정해진 시간에 NotificationManager 클래스로 runNoticeCheck(productList)를 요청한다. NotificationManager는 checkTargetPrice(productId, curPrice, targetPrice)를 실행하고, Product 클래스의 checkTarget(productId, curPrice, targetPrice)를 통해서 현재 가격이 목표 가격 이하인지 확인한다. <br>

1. 현재 가격이 목표 가격 이하인 경우, NotificationManager는 checkDuplicateNotice(productId, curPrice)를 실행해 같은 가격에서 이미 알림이 전송되었는지 확인한다.
2. 중복 알림이 아닌 경우, NotificationManager는 createNotice(productId, NuserId, message)를 실행해 알림을 생성한다. 이후 Notification 클래스에 새 알림 정보를 저장하고 sendNotice(notificationId)와 sendEmailNotice(NuserId, message)를 실행해 웹, 이메일 알림을 전송한다.
3. 알림 전송에 성공한 경우, Scheduler에 알림 전송 완료 결과를 반환한다.
4. 알림 전송에 실패한 경우, NotificationManager는 ErrorLog 클래스에 savelog("알림 전송 실패")를 요청해 오류 내용을 저장하고 Scheduler에 알림 전송 실패 처리 완료 결과를 반환한다.
5. 동일 가격에서 이미 알림이 전송된 경우, 중복 알림을 막기 위해 새 알림을 생성하지 않는다.
6. 현재 가격이 목표 가격보다 높은 경우, 알림 조건을 만족하지 않아 알림을 생성하지 않는다.

## 4. State machine diagram

### 1) Login State Machine Diagram
<img width="932" height="344" alt="image" src="https://github.com/user-attachments/assets/15ebc65d-268a-4b92-aac5-df63ceef85ad" />
<br>

Login State Machine Diagram는 사용자가 시스템에 접속하는 과정에서 로그인 화면이 어떻게 전환되는지를 보여준다. Login Page 상태에서 로그인 화면을 보여주고 사용자가 로그인 버튼을 누르면 Checking Input 상태로 넘어간다. 이때는 사용자가 입력한 아이디와 비밀번호란이 비어 있는지 확인한다. <br>

입력값이 비어있을 땐 Login Failed 상태로 이동해 로그인 실패 메세지를 출력하고 사용자를 다시 Login Page로 돌려보내 재로그인을 시도할 수 있게 한다. 입력값이 비어있지 않은 경우는 Checking Account 상태로 이동해 입력한 정보가 등록된 사용자 또는 관리자 정보와 일치하는지 확인한다. <br>

계정 정보가 일반 사용자와 일치하면 User Main Page 상태로 이동해 사용자 메인 화면을 보여준다. 관리자 계정과 일치하는 경우 Admin Log Page 상태로 이동해 관리자 로그 화면을 보여준다. 계정 정보가 일치하지 않는 경우는 Login Failed 상태로 이동해 로그인 실패 메세지를 출력하고 다시 로그인 화면으로 돌아갈 수 있게 한다.

### 2) Product State Machine Diagram
<img width="1354" height="691" alt="image" src="https://github.com/user-attachments/assets/909b4bba-40ec-4282-9e3b-aa0862679194" />
<br>

Product State Machine Diagram은 상품이 등록된 뒤 가격 추적, 목표가 도달, 수집 실패, 품절, 목표 가격 변경, 삭제 상태로 전환되는 과정을 보여준다. 처음 상품 등록이 성공하면 Registered 상태가 되고 상품 정보를 저장한다. 이후 가격 추적이 시작되면 Tracking 상태로 이동하고, 시스템은 다음 가격 확인 시간을 기다린다.
<br>

1. 가격 확인 시간이 되면 Tracking 상태에서 Checking Price 상태로 이동해 상품의 현재가를 확인한다. 현재 가격이 목표 가격보다 높으면 다시 Tracking 상태로 돌아가서 다음 가격 확인 시간까지 기다린다.
2. 현재 가격이 목표 가격에 도달하면 Target Reached 상태로 간다. 이때는 목표 가격 도달 상태를 표시하고 알림 처리과정이 끝나면 다시 Tracking 상태로 돌아가 다음 가격 확인 시간을 기다린다.
3. 가격 확인 과정에서 수집을 실패하면 Collection Failed 상태로 이동해 가격 수집 실패 상태를 표시하고 다음 가격 확인 시간을 기다리기 위해 Tracking 상태로 돌아간다.
4. 상품이 품절된 경우 Sold Out 상태로 이동해 품절 상태로 변경하고 다음 가격 확인 시간까지 기다리다 Tracking 상태로 돌아간다.
5. 사용자가 목표 가격을 변경하면 Changing Target Price 상태로 이동해 변경된 목표 가격을 저장하고 다시 Tracking 상태로 돌아가서 가격 추적을 이어서 한다.
6. 사용자가 상품을 삭제하면 Deleted 상태로 이동해 상품 정보를 삭제하고 가격 수집을 종료한다.
<br>
<br>

### 3) NotificationState Machine Diagram
<img width="782" height="797" alt="image" src="https://github.com/user-attachments/assets/0edeec5b-2abc-4ec5-b6e0-54beb61a4706" />
<br>

Notification State Machine Diagram은 목표 가격 도달 후 알림이 생성, 전송되고 사용자가 알림을 읽음 처리하는 과정을 보여준다. 처음에는 Notification Waiting 상태에서 기다리다 상품의 현재 가격이 목표 가격에 도달하면 Checking Duplicate 상태로 이동해 같은 가격에서 이미 알림이 생성되었는지 확인한다. <br>

1. 이미 동일한 가격에서 알림이 생성된 경우, Not Created 상태로 이동해 중복 알림 생성을 막기위해 새 알림을 만들지 않고 알림 처리 과정을 종료한다.
2. 중복 알림이 아닌 경우 Created 상태로 이동해 새 알림 정보를 저장하고 저장이 완료되면 Sending 상태로 이동해 웹 알림과 이메일 알림을 전송한다.
3. 알림 전송에 성공하면 Unread 상태가 된다. 이때는 사용자가 아직 확인하지 않은 알림으로 표시된다.
4. 사용자가 알림을 직접 확인하거나 모두 읽음 처리를 하면 Read 상태로 이동해 해당 알림이 읽음 상태로 표시된다.
5. 알림 전송에 실패한 경우, Send Failed 상태로 이동해 이후 Error Logged 상태에서 알림 전송 실패 내용을 오류 로그로 저장하고 알림 처리 과정을 종료한다.

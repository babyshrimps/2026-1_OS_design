# 1. Business purpose  
![DT_1KE10041](https://github.com/user-attachments/assets/1fa5ea53-33d1-4c17-abb9-19c50d8bfafc)  
[그림 1]  
  
 인터넷이 빠르게 발전하면서 전자상거래가 우리의 일상에 자리 잡게 되었다. 이에 따라서 사람들은 오프라인 매장을 직접 방문하기 보다는 온라인 쇼핑몰, 모바일 앱을 통해 구매하는 경우가 증가하였고 다양한 쇼핑 플랫폼의 등장에 의해 소비자는 이전보다 다양한 상품과 가격 정보를 손쉽게 비교할 수 있게 되었다. 그리고 각 플랫폼에서도 경쟁사를 의식해 가격 경쟁을 벌이는 경우가 많아졌다. 그 결과 소비자는 보다 합리적이고 현명한 소비를 하기 위해서 가격 비교를 필수적으로 고려하게 되었다.  
 이 프로젝트는 이런 온라인 쇼핑 과정에서 소비자(사용자) 가 지속적으로 상품 가격을 확인해야 하는 불편함을 줄이기 위해 기획되었다. 일반적으로 사용자들은 마음에 드는 상품을 발견하면 바로 구매하지 않고 이벤트 기간이나 가격 인하를 기다리는 경우가 많다. 그러나 이 과정에서 사용자는 같은 상품 페이지를 여러 번 재방문 해야하고 언제 가격이 내려갔는지 직접 확인해야 한다.  
  이러한 문제를 해결하기 위해 사용자가 관심 있는 상품을 시스템에 등록한 뒤, 원하는 가격 조건을 설정하면 시스템이 주기적으로 해당 상품의 가격을 확인하고 설정 가격 이하로 내려갔을 때 알림을 제공하는 웹 기반 서비스를 제공할 예정이다. 또한 가격 이력을 저장해 사용자가 가격 변동 흐름을 파악할 수 있게 할 것이다.

# 4. Concept of operation  

1) 회윈가입 및 로그인

| Purpose | 사용자별 관심 상품과 알림 설정을 개별적으로 관리한다 |
| Approach | 회원가입 후 ID와 비밀번호를 입력하면 서버가 회원 정보를 확인하여 로그인 성공 여부를 반환한다 |
| Dynamics | 사용자가 최초 이용하거나 로그인 기능을 사용할 때 동작한다 |
| Goals | 사용자 인증 기능 구현 |
  
2) 상품 등록

| Purpose | 사용자가 원하는 상품을 가격 추적 대상으로 등록한다 |
| Approach | 상품 링크를 입력하면 시스템이 해당 상품 정보를 저장한다 |
| Dynamics | 상품 등록 화면에서 등록 버튼을 누를 때 동작한다 |
| Goals | 추적 대상 상품을 시스템에 등록 |
  
3) 목표 가격 설정

| Purpose | 사용자가 원하는 가격 알림 기준을 설정한다 |
| Approach | 사용자가 특정 금액을 입력하면 해당 값을 목표 가격으로 저장한다 |
| Dynamics | 상품 등록 직후 또는 상품 설정 변경 시 동작한다 |
| Goals | 조건 기반 알림의 기준 마련 |
  
4) 관심 상품 목록 조회

| Purpose | 사용자가 자신이 등록한 상품의 현재 상태를 확인한다 |
| Approach | 서버가 사용자 계정에 연결된 상품 목록과 최신 가격, 목표 가격, 상태 정보를 반환한다 |
| Dynamics | 사용자가 대시보드 또는 마이페이지를 열 때 동작한다 |
| Goals | 개인별 관리 화면을 제공 |
  
5) 가격 정보 수집

| Purpose | 등록된 상품의 최신 가격을 가져온다 |
| Approach | 시스템이 일정 시간마다 링크를 타고 들어가 쇼핑몰에서 상품 가격을 조회한다 |
| Dynamics | 스케줄러에 의해 자동 실행한다 |
| Goals | 최신 가격 데이터 수집 |
  
6) 가격 이력 저장

| Purpose | 가격 변동 과정을 기록한다 |
| Approach | 수집된 가격을 수집 시각과 함께 데이터베이스에 저장한다 |
| Dynamics | 가격 수집이 완료될 때마다 동작한다 |
| Goals | 가격 이력 데이터 누적 관리 |
  
7) 조건 충족 여부 판별

| Purpose | 현재 가격이 사용자의 목표 가격 조건에 맞는지 확인한다 |
| Approach | 시스템이 현재 가격과 목표 가격을 비교하여 조건을 만족하는지 계산한다 |
| Dynamics | 가격 수집 직후 동작한다 |
| Goals | 알림 발송 여부 결정 |
  
8) 알림 발송

| Purpose | 사용자가 가격 하락 시점을 즉시 알 수 있도록 한다 |
| Approach | 조건이 충족되면 시스템이 이메일 또는 웹 알림을 생성하여 사용자에게 전달한다 |
| Dynamics | 목표 가격 이하가 되었을 때 동작한다 |
| Goals | 가격 알림 기능 구현 |
  
9) 가격 추이 조회

| Purpose | 사용자가 가격 변동 패턴을 확인하여 구매 시점을 정할 수 있게 한다 |
| Approach | 저장된 가격 이력 데이터를 표 또는 그래프 형태로 제공한다 |
| Dynamics | 사용자가 상품 상세 화면을 조회할 때 동작한다 |
| Goals | 가격 변화 흐름을 시각적으로 제공 |
  
10) 알림 내역 조회

| Purpose | 사용자가 언제, 어떤 조건으로 알림을 받았는지 확인한다 |
| Approach | 서버가 사용자별 알림 발송 기록을 조회해 목록으로 제공한다 |
| Dynamics | 사용자가 알림 페이지를 열 때 동작한다 |
| Goals | 알림 기록 관리 |
  
11) 상품 삭제 또는 설정 변경

| Purpose | 사용자가 필요에 따라 추적 상품을 제거하거나 기준 가격을 수정한다 |
| Approach | 사용자가 삭제 또는 수정 버튼 누르면 서버가 상품 상태 또는 목표 가격을 업데이트 한다 |
| Dynamics | 관심 상품을 관리할 때 동작한다 |
| Goals | 사용자 편의성과 관리 기능 높임 |

12) 오류 및 로그 확인

| Purpose | 가격 수집 실패, 알림 실패, 예외 상황을 점검한다 |
| Approach | 관리자가 로그 화면에서 실패 내역과 시스템 동작 기록을 확인한다 |
| Dynamics | 오류가 생기거나 관리자 점검 시 동작한다 |
| Goals | 시스템 안정성과 유지보수성 확보 |

# 5. Problem statement  
**Overview**  
Price Tracking secretary는 사용자가 원하는 물건의 목표가를 설정하면 시스템이 설정 가격 이하임을 확인했을 때 사용자에게 알림을 보내주는 서비스이다. 하지만 이를 구현하는 과정에서 몇 가지 문제점이 발생할 수 있다.  

1. 외부 쇼핑몰의 가격 정보를 수집하기 힘들다. 쇼핑몰마다 상품 페이지의 구조, 가격 표시 방식 등을 구현한 방법이 다르기 때문에 동일한 방법으로 모든 상품 정보를 처리하기 어렵다. 또한 정가, 할인가, 쿠폰 적용가 등이 따로 표기되어 있어 실제 구매가를 판단하는데 문제가 발생할 수 있다. 이를 해결하기 위해 범위를 일부 쇼핑몰로 제한해서 기능을 구현하고, 수집한 데이터를 같은 형식으로 변환하여 저장해 시스템이 가격 정보를 일관되게 관리할 수 있도록 한다.  
  
2. 가격 정보 백업 설정이다. Price Tracking secretary는 단순히 현재 가격만 보여주는 것이 아니라 가격이 어떻게 변해 왔는지도 제공한다. 그러나 가격 값이 지속적으로 누적되면 데이터 양이 증가하고, 이를 조회하는 과정에서도 효율적인 구조가 필요하다. 또한 너무 늦게 수집하면 정보의 신뢰성이, 너무 자주 수집하면 서버 자원의 사용량이 증가해 시스템 효율이 저하될 것이다. 따라서 가격 정보를 날짜와 시간 기준으로 저장하고, 사용자가 필요한 범위의 데이터만 조회할 수 있도록 하려 한다 (가격 이력은 사용자가 상품을 시스템에 등록할 때 부터 수집, 저장할 예정이며 등록 이전의 정보는 별도의 데이터 입력이 없으면 제공하지 않는다). 또한 이를 표나 그래프 형태로 제공해 사용자가 가격 변동 흐름을 한 눈에 알아보기 쉽도록 할 계획이다. 가격 수집 간격은 1~2시간 간격으로 진행할 예정이다.
  
3. 중복 알림 발생이다. 사용자가 설정한 목표 가격 이하 상태가 일정 시간 이상 유지될 경우, 동일한 상품에 대해 같은 알림이 반복적으로 전달될 가능성이 있다. 따라서 알림 발송 이력을 저장하고, 동일한 가격에서 이미 알림이 발송된 경우에는 가격이 새로 바뀌기 전까지는 발송을 막는 방식으로 중복을 줄이고자 한다.  
  
4. 예외 상황 처리 이다. 시스템을 이용하다 보면 상품이 삭제되거나, 품절 상태가 되거나, 사용자가 잘못된 URL을 입력하는 등의 상황이 발생할 수 있다. 이러한 경우 시스템 오작동으로 인한 혼란이 생길 수 있기 때문에 가격 수집 실패 시 오류 내용을 기록하고, 해당 상품의 상태를 사용자에게 알려주는 방향으로 설계한다. 이를 통해 사용자가 현재 어떤 문제가 발생했는지를 확인할 수 있다.

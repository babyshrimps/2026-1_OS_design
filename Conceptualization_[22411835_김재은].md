### 1. Business purpose  
![DT_1KE10041](https://github.com/user-attachments/assets/1fa5ea53-33d1-4c17-abb9-19c50d8bfafc)  
[그림 1] 년도별 온라인 쇼핑몰 거래액 합계  
  
 인터넷이 빠르게 발전하면서 전자상거래가 우리의 일상에 자리 잡게 되었다. 이에 따라서 사람들은 오프라인 매장을 직접 방문하기 보다는 온라인 쇼핑몰, 모바일 앱을 통해 구매하는 경우가 증가하였고 다양한 쇼핑 플랫폼의 등장에 의해 소비자는 이전보다 다양한 상품과 가격 정보를 손쉽게 비교할 수 있게 되었다. 그리고 각 플랫폼에서도 경쟁사를 의식해 가격 경쟁을 벌이는 경우가 많아졌다. 그 결과 소비자는 보다 합리적이고 현명한 소비를 하기 위해서 가격 비교를 필수적으로 고려하게 되었다.  
 이 프로젝트는 이런 온라인 쇼핑 과정에서 소비자(사용자) 가 지속적으로 상품 가격을 확인해야 하는 불편함을 줄이기 위해 기획되었다. 일반적으로 사용자들은 마음에 드는 상품을 발견하면 바로 구매하지 않고 이벤트 기간이나 가격 인하를 기다리는 경우가 많다. 그러나 이 과정에서 사용자는 같은 상품 페이지를 여러 번 재방문 해야하고 언제 가격이 내려갔는지 직접 확인해야 한다.  
  이러한 문제를 해결하기 위해 사용자가 관심 있는 상품을 시스템에 등록한 뒤, 원하는 가격 조건을 설정하면 시스템이 주기적으로 해당 상품의 가격을 확인하고 설정 가격 이하로 내려갔을 때 알림을 제공하는 웹 기반 서비스를 제공할 예정이다. 또한 가격 이력을 저장해 사용자가 가격 변동 흐름을 파악할 수 있게 할 것이다.

### 5. Problem statement  
**Overview**  
Price Tracking secretary는 사용자가 시스템에 구매하길 원하는 상품의 링크를 저장한 뒤 구매 희망가를 설정하면 시스템이 상품의 가격을 지속적으로 체크 및 저장하고 희망가 이하임을 확인했을 때 사용자에게 알림을 보내주는 서비스이다. 하지만 이를 구현하는 과정에서 몇 가지 문제점이 발생할 수 있다.  

첫번째로 외부 쇼핑몰의 가격 정보를 수집하기 힘들다. 현재 상태에서 실제 외부 쇼핑몰의 데이터를 멋대로 크롤링할 경우, 법적으로 문제가 발생할 수 있다. 따라서 현재 프로그램의 경우 상품 링크는 사용자가 등록한 상품을 확인하기 위한 참고 정보로 저장하고, 가격 추적 기능은 법적 문제를 피하기 위해 모의 가격 데이터를 생성하는 방식으로 구현한다. (실제 서비스로 확장하는 경우 각 쇼핑몰의 공식 API 제공 여부와 이용약관 등을 확인하고 허용을 받은 뒤 정보를 연동해야한다.)
  
두번째, 가격 정보 백업 설정이다. Price Tracking secretary는 현재 가격만 보여주는 것이 아닌 과거 가격 내역도 함께 제공하는데 가격 값이 지속적으로 누적되면 데이터 양이 증가하고, 이를 조회하는 과정에서도 효율적인 구조가 필요하다. 또한 너무 늦게 수집하면 정보의 신뢰성이, 너무 자주 수집하면 시스템 자원의 사용량이 증가해 시스템 효율이 저하될 것이다. 따라서 가격 정보를 날짜와 시간 기준으로 저장하고, 사용자가 필요한 범위의 데이터만 조회할 수 있도록 하려 한다 (가격 이력은 사용자가 상품을 시스템에 등록할 때 부터 수집, 저장할 예정이며 등록 이전의 정보는 별도의 데이터 입력이 없으면 제공하지 않는다). 또한 이를 표나 그래프 형태로 제공해 사용자가 가격 변동 흐름을 한 눈에 알아보기 쉽도록 할 계획이다. 이번 프로젝트에선 기능 구현을 체크하기 위해 1분마다 진행한다.
  
세번째, 중복 알림 발생이다. 상품의 현재 가격이 사용자가 설정한 목표 가격 이하인 상태로 일정 시간 유지되는 경우, 가격 알림이 반복적으로 전달될 가능성이 있다. 따라서 알림 발송 이력을 저장하고, 동일한 가격에서 이미 알림이 발송된 경우에는 가격이 새로 바뀌기 전까지는 발송을 막는 방식으로 중복을 줄이고자 한다.  
  
마지막으로 예외 상황 처리이다. 시스템을 이용하다 보면 상품이 삭제되거나, 품절 상태가 되거나, 사용자가 잘못된 URL을 입력하는 등의 상황이 발생할 수 있다. 이러한 경우 시스템 오작동으로 인한 혼란이 생길 수 있기 때문에 가격 수집 실패 시 오류 내용을 기록하고, 해당 상품의 상태를 사용자에게 알려주는 방향으로 설계한다. 이를 통해 사용자가 현재 어떤 문제가 발생했는지 확인할 수 있다.

# 6. Glossary  

| Terms | Description |
|---|---|
| 가격 수집 | 등록된 상품의 현재 가격을 모의 데이터로 갱신하고 가격 이력에 저장하는 과정 |
| 가격 이력 | 시간에 따라 저장된 상품 가격 기록 |
| 가격 추적 | 상품 가격을 일정 간격으로 확인하면서 변동 여부를 확인하는 기능 |
| 관리자 | 시스템의 오류나 로그를 확인하고 서비스 상태를 점검하는 운영 주체 |
| 관심 상품 | 사용자가 가격 변화를 확인하기 위해 등록해 둔 상품 |
| 로그 | 시스템 실행 기록이나 오류 내용을 남겨 두는 데이터 |
| 마이페이지 | 사용자가 등록한 상품을 모아서 확인 가능하며, 이외 알림 설정 등을 만질 수 있는 페이지 |
| 목표 가격 | 사용자가 알림을 받고자 직접 설정한 기준 금액 |
| 사용자 | 상품을 등록하고 목표 가격을 설정하며, 가격 변동을 확인하고 알림을 받는 서비스 이용자 |
| 상품 | 사용자가 가격 추적 대상으로 등록하는 물건 |
| 상품 링크 | 사용자가 시스템에 입력하는 상품 페이지의 주소 |
| 수집 주기 | 시스템이 가격 정보를 다시 확인하는 시간 간격 |
| 스케줄러 | 정해진 시간마다 가격 확인 작업을 실행하고 알림을 보내는 기능 |
| 시스템 | 사용자 정보, 상품 정보, 가격 기록, 알림 내역 등을 저장하고 전반적인 업무를 처리 |
| 알림 | 현재 상품의 가격이 목표가보다 작거나 같아진 경우 사용자에게 전달되는 안내 메시지 |
| 예외 상황 | 품절, 링크 오류, 가격 수집 실패처럼 정보가 정상적으로 처리되지 않는 경우 |
| 외부 쇼핑몰 | 시스템 밖에서 실제 상품 정보와 링크를 제공하는 온라인 쇼핑 사이트 |
| 웹 알림 | 웹페이지 안에서 바로 확인할 수 있는 알림 방식 |
| 이메일 알림 | 사용자의 이메일 주소로 전달되는 알림 방식 |
| 현재 가격 | 시스템이 가장 최근에 확인한 상품의 가격 |
